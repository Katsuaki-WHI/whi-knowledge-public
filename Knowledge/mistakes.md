# mistakes.md - 再発防止の核

かつさんから明示的に訂正された、再発しうるパターンを記録。
新規追加は AI-RULES.md の「3. mistakes.md への追記ルール」3条件を満たすときのみ。

---

## 2026-05-19：Build PASS は本番OKを意味しない

**NG**：`npm run build` が成功したら「実装完了」と報告する
**Correct**：ビルド成功後、必ずブラウザベースの実HTTPリクエストフローで動作確認してから「実装完了」と報告する
**Trigger**：Next.js/RSC（React Server Components）を使うプロジェクトでの実装完了報告時。
RSCマーシャリング違反はビルド成功でも本番でエラーになるため、ビルド成功だけでは品質を担保できない。

## 2026-05-19：「調査済み・正常」報告を鵜呑みにしない

**NG**：Claude Codeが「動作確認しました、正常です」と報告したら、それで完了扱いにする
**Correct**：ブラウザベースの実HTTPリクエストフローで、かつさんもしくはClaude Code自身が実際に画面・APIを叩いて確認する
**Trigger**：実装完了報告・バグ修正完了報告のとき。
過去に「正常」報告のあと本番で再現した事例あり。

## 2026-05-19：APIエラー時の連打でコストが跳ねる

**NG**：APIエラー発生時、原因究明前に再実行を繰り返す
**Correct**：エラー発生時はまず原因究明。Spend Limit設定とアラート閾値を必ず設定。連打しない。
**Trigger**：Anthropic API・OpenAI APIなど従量課金APIのデバッグ時。
過去に予期せぬコスト消費が発生した。詳細は [[api-cost-management]]。

## 2026-05-20：マスター移行時は旧参照を全使用箇所grepで総点検する

**NG**：マスターデータを新旧で移行するとき、一部のセクション・コンポーネントだけ
旧マスター参照を残す
**Correct**：移行時は旧参照（旧関数名・旧配列名・.find()等）を src/ 全体で grep し、
全使用箇所を一覧化してから移行する。移行漏れがあると、旧マスターに無い値を引いた
特定ユーザーだけ表示が欠ける（他ユーザーは正常なので気づきにくい）。
**Trigger**：マスターデータの新旧移行時・getXxx系の参照関数を差し替えるとき。
**実例**：なかまクエスト「お主のキャラクター」セクションが getCharacter()（旧21件マスター）
を参照したままで、C_貢献（新126件マスターのみの値）を持つメンバーだけカードが空白になった。
原因はデータ欠損ではなくコードの移行漏れ。getCharacterById()（126件対応）へ切替で解決。

## 2026-05-27：マスターのキー命名規則は1つに統一する（4分裂が同種バグを量産）

根本原因はマスターキー命名規則の4分裂。

## 2026-05-29：Supabaseには2層のタイムアウトがある（pg_cron＋timeout延長で回避）

**NG**：大規模集計を SQL Editor で実行し「upstream timeout（~60秒）」、pg_cron に逃がしても「きっかり2:00で失敗」を繰り返す
**Correct**：Supabaseのタイムアウトは**2層・別レイヤー**と認識する。①ゲートウェイ ~60秒（SQL Editorの HTTP層）→**pg_cron等**で回避。②DB statement_timeout 2分→**関数ヘッダの `SET statement_timeout TO '600s'`**（必要なら `alter role postgres set statement_timeout`）で回避。**大規模集計は最初から pg_cron＋timeout延長を前提に設計**する。SQL Editor は短いクエリ専用と割り切る
**Trigger**：Supabaseで数分かかる集計・データ移行・MV REFRESH等を実行するとき
**出典**：SQWサーベイ2 統合設計 段階D（2026/05/29）。詳細は [[supabase-statement-timeout-layers]]・[[supabase-long-running-aggregation-pattern]]

## 2026-06-15：Supabaseは「組織単位」課金（プロジェクト単位ではない）

- 1組織に複数プロジェクトを置いた状態で組織をProにすると、全プロジェクトが課金対象になる。Pro料金は1プロジェクト目に加え、追加プロジェクトごとに定額が加算されて積み上がる。

## 2026-06-27：メール送信サービスは「アカウント／APIキー／ドメイン」が別物

**NG**：「ドメインが Verified なら送れる」と思い込み、本番が使う APIキーがどのアカウント（ワークスペース）のものかを確認せず、別アカウントのキーから送って 403（domain not verified）になる。
**Correct**：メール不達（403 not verified）時は、まず「本番が使うキーが、そのドメインを検証したアカウントのキーか」を確認する。送信サービス（Resend等）は ①アカウント ②APIキー（アカウントに属す）③ドメイン（アカウント単位で検証）が別物。ドメインが Verified でも別アカウントのキーから送れば 403。キーで `GET /domains` が空配列を返すなら、そのキーのアカウントにドメインが無い＝アカウント不一致が真因。
**Trigger**：本番メールが送れない／送信APIで 403 not verified が出るとき。送信元ドメインを変えるとき。
