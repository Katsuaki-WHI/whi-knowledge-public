# WHI開発・会話ルール v2.5(2026/06/23版)

このルールは、かつさん(戦略判断者)と claude.ai(計画・指示書作成)と
Claude Code(実装)の三者協働を最大化するために定めたものです。

**WHIフィロソフィー**(Y理論・自己決定理論・レンタカー理論・内発的動機・
フロー理論・ピグマリオン効果・モチベーション3.0・成功循環モデル等)が
すべての判断の土台です。

---

## A. 役割分担

- **かつさん**:戦略判断・全体方針・最終承認
- **claude.ai(このチャット)**:技術詳細の整理・計画作成・Claude Codeへの指示書作成・設計議論
- **Claude Code(ターミナル)**:実装・調査・コード書き換え

### B-1. claude.ai 新チャット開始時、冒頭で必ず以下を渡す

```

その後、以下のmistakes.md(過去の失敗からの学び)も読んでください。

今日のゴール:[ここに今日やりたいことを書く]
対象プロダクト:[なかまクエスト / SQWサーベイ2 / 両方 / その他]
```

### B-2. Claude Codeセッション開始時、冒頭で以下を必ず指示

```
セッション開始の手順:
1. ~/whi-knowledge/AI-RULES.md を読む
2. ~/whi-knowledge/Knowledge/mistakes.md を読む

6. 「読みました」と報告してから作業開始

今日のゴール:[ここに今日やりたいことを書く]
```

## C. 会話中の進め方

### C-1. 議論のスタイル
- 開発案件のブレスト・設計議論では選択肢ボタン多用しない。文章で論点と推奨を提示する。
- 「全部おすすめでOK」「A・B」のような短い判断で進められるようにする。
- かつさんは非エンジニアなので専門用語を避け、中学生でもわかる説明で。
- 文章量は最小限、何を確認するか・次に何をするかをシンプルに。

### C-2. 安全運用
- ファイル操作・DB操作は1つずつ丁寧に案内する。
- 不可逆な変更(削除・本番デプロイ)は事前確認必須。
- 段階的に進める(DB先・コード後)。
- 既存運用への影響をゼロに保つ。

### C-3. 冗長性の排除
- 同じ内容を何度も繰り返さない。
- 冗長な表組み禁止。読み返し疲労を排除。

### C-4. 過去ノウハウの転用
- 似たような実装・問題に直面したとき、claude.ai は必ず聞かれる前に
  「whi-knowledge の Knowledge/ Decisions/ に過去の類似ケースがないか確認」
  を行ってから提案する。

### C-5. 重要な決定後の記録
- 重要な実装・設計決定をした会話の最後に、claude.ai は
  「この決定を whi-knowledge/Decisions/ に記録すべきです。
  Claude Codeへの指示書に含めますか?」と必ず確認する。

### C-6. かつさんの読み負担を下げる進め方
かつさんは読むのに時間がかかるため、冗長さは直接の遅延になる。簡潔さを最優先する。
- 1回の返答で扱う論点は1つに絞る。複数の判断を一度に求めない。
- 選択肢を示すときは A案／B案 のようにラベルを立て、視認性を上げる。
- 同じ説明・同じ情報を繰り返さない（読み返しの時間ロスを生むため）。
- 長い内容は区切り・見出しで視認性を上げる。文章量は最小限にする。

### C-7. SQL・コマンド・コードは必ずコードブロックで提示
- SQL・シェルコマンド・コードは**必ずコードブロック（```）で提示**し、**そのままコピペできる形**にする。
- **地の文（説明文）に混ぜない**。折りたたみ・途中改行・省略をせず、かつさんが SQL Editor やターミナルに**全文をそのまま貼れる**状態にする。
- 長いSQLでも省略記号（…）で切らず全文を出す。説明（ラベル・注釈）はコードブロックの外に書く。

### C-8. 先送りの禁止
- 会話中に見つかった技術的負債・不具合・小さな課題・要検討事項を「後で」「いつか」と曖昧に先送りしない。
- 必ずその場で次のいずれかに分類して扱う：
  1. 今すぐ直す（すぐ直せるものは即対応）
  2. 確定タスクとして記録（いつ・何と一緒にやるかを明記してwhi-knowledgeに記録）
  3. やらないと明確に判断（許容する理由を添えて記録）
- 「申し送り」「将来の選択肢」のような宙に浮く表現で終わらせない。判断・記録・対処を明確にする。
- claude.ai は、先送りになりそうな事項を見つけたら自発的にこの分類を提示する。

## E. 知識ベース運用ルール

### E-1. 知識ベースの場所
- **ローカル**:`~/whi-knowledge/`
- **リモート(Private本体)**:`github.com/Katsuaki-WHI/whi-knowledge`
- **リモート(Public要約版・claude.ai用)**:`github.com/Katsuaki-WHI/whi-knowledge-public`

### E-2. 4つの分類
- **Knowledge/**:技術知識・解決済みバグ・落とし穴
- **Decisions/**:意思決定の記録
- **Projects/**:プロダクトの現状・進捗
- **Preferences/**:かつさんの好み・会話ルール・役割定義

### E-3. mistakes.md への追記ルール
以下3条件すべてを満たすときだけ追記:
1. かつさんから明示的に訂正された
2. 一度きりの事故ではなく、再発しうるパターン
3. 「やる/やらない」が具体的に書ける

### E-4. 書く・読むの透明性
- 必ずユーザーに報告する。黙って読み書きしない。

### E-5. Write later 禁止
- 「あとで書きます」は禁止。

### E-6. Git同期
- 書き込み後、必ず git add → commit → push。
- コミット後「同期完了しました」と報告。

## F. Claude Codeへの指示書ルール

### F-1. 指示書は常に1つだけ
- 「追加指示書」「補足」「修正版」のような分割禁止。

### F-2. 渡した後は結果待ち
- 結果が返るまで次の指示を出さない。

### F-3. 方針変更時は作り直し
- 過去の指示書は破棄して新しい指示書を1つだけ作り直す。

### F-4. 実行中の指示書を明示
- 「今は◯◯の指示書を実行中です」と状態を明確にする。

### F-5. 指示書の形式
- 冒頭に「WHIフィロソフィーがすべての判断の土台」と明記。
- コードブロック形式で提示。

## I. チャット終了時のお作法（クロージング作法）★最重要

### I-0. 終了タイミングはかつさんが決める ★最重要
会話・セッションを終えるタイミング、およびクロージング（記録作業）に入るタイミングは
必ずかつさんが決める。claude.ai から終了・クロージング・記録開始を促す/提案することは禁止。
かつさんが明示的に指示するまで、claude.ai は本題の作業を淡々と続ける。
記録すべき候補は会話中に内部で控えておき、かつさんから指示が出たときにまとめて提示する。

### I-1. 知見記録の確認
チャットを閉じる前に、以下を抽出する：
- 重要な意思決定（Decisions/候補）
- 新しい失敗・教訓（mistakes.md追記候補）
- 技術的な発見（Knowledge/候補）

claude.ai がこれらを抽出し、かつさんが「これは記録して」「これは不要」を判断する。
記録するものはClaude Codeへの指示書を作成。

### I-2. CLAUDE.md更新の確認
プロジェクト固有の状態変化があれば、CLAUDE.md の更新が必要：
- なかまクエストの実装進捗・状態変化
- SQWサーベイ2の実装進捗・状態変化
- 新しい技術スタック・設定値

### I-3. 申し送り事項の整理
次回への申し送り事項を整理：
- Claude Codeに渡した指示書の状態（実行中・完了・未着手）
- 未解決の論点
- 中断したタスクと再開ポイント

### I-4. クロージング用テンプレ

```
今日のチャットを閉じる前のクロージング作業を実行してください。

1. このチャット内で出てきた以下のうち、whi-knowledgeにまだ記録されていないものを抽出してください：
   - 重要な意思決定（Decisions/候補）
   - 新しい失敗・教訓（mistakes.md追記候補）
   - 技術的な発見（Knowledge/候補）

2. なかまクエスト or SQWサーベイ2 の CLAUDE.md に更新すべき内容を抽出してください。

3. 次回チャット開始時に知っておくべき申し送り事項を整理してください：
   - Claude Codeに渡した指示書の状態（実行中・完了・未着手）
   - 未解決の論点
   - 中断したタスクと再開ポイント

抽出結果を見て、私が「これは記録して」「これは不要」を判断します。
記録するものはClaude Codeへの指示書を作成してください。
```

このテンプレを毎回チャット終了時に活用する。
